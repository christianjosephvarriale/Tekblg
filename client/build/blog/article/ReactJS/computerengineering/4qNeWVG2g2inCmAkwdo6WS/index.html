ReactJS best practices

  • 2020-01-22T06:31-05:00
  • By Tekblg

Let's fix some real bugs here

If you're a programmer and you're new to ReactJS or just coding in general, it's very important to make sure you get the basics right.

Unfortunately the basics of algorithms / data structures, syntax, and problem solving capabilities are although important and translational, they are not indicative of your "style" as a programmer.

Here's something to think about

What is Bad code exactly? It might be code that doesn't work in the obvious case, however in the less obvious case it may be code which is prone to errors, or worse is impossible to debug.

Let's go over examples of both good and bad code. This is code that's both satirical but does actually mimic some of the code we've seen in industry

Unreadable Comments

This one's controversial because comments can really help to clarify the functions behind complex code and can also provide extensive documentation on inputs and outputs of functions ( which should be done ).

const fizzBuzz = () => {
   // iterate from 1 to 20
   for (var i=1; i <= 20; i++) { 
   // check to see if the iterator is divisible by both 5 and 3
   if (i % 15 == 0) {
      // if the iterator is divisible by both 5 and 3 then console.log("FizzBuzz") 
      console.log("FizzBuzz"); 
   // check to see if the iterator is divisible by only 3
   } else if (i % 3 == 0) { 
      // if the iterator is divisible by only 3 then console.log("Fizz"); 
      console.log("Fizz"); 
   // check to see if the iterator is divisible by only 5
   } else if (i % 5 == 0) { 
      // if the iterator is divisible by only 5 then console.log("Buzz"); 
      console.log("Buzz"); 
   // the iterator is not divisible by either 5 or 3
   } else { 
      console.log(i); 
   } 
}
EOC

While it's true for complicated operations it's important to have a comment, especially if the operation could be further encapsulated into a seperate, smaller function ( which you should do anyways if the sub-task isn't overly specific ), let's not blow our brains out. Comments which reiterate basic concepts which readable code could communicate is distasteful and it necessarily inflates the size of your code and thereby it's readability. Here's however a good template to follow when commenting functions. In addition, if you follow the format below VS code will automatically template it to look professional on hover when looking at function definitions. It's worth standardizing across your application.

/**
* Des: here's a short description of the function's purpose
* @param foo - very short des of parameter and it's type 
* ...
* @returns bar - very short des of return value and it's type
*/
EOC

because the function is littered with numbers as opposed to global constants or variable names, it's unclear what the algorithm or computation is. Variables names and comments can give you a hint, but our goal here is to be self documenting by having a naming convention for variables. For reactJS using camelCase on ALL variables is key. camelCase means that instead of using spaces between words we capitalize all words succeeding the first word to indicate separation.

The other convention is for variable names it's important to make them detailed, specific, but as concise as possible. This comes with experience but names like postsLst is preferred over postsInFeaturedBlogSection and is also preferred over something lacking detail such as items. Another controversial but widely used idea is to abbreviate words by taking out letters to preserve it phonetically but minimize the amount of typing. For example lst => list, arr => arry, object => obj and so on.
