A general guide to Unit Testing in Node.JS

  • 2020-01-28T16:09+02:00
  • By Tekblg

You're gonna love this one

Testing is great. No seriously it is. It adds more development time before you can finally tell move that pesky component to the QA phase on your scrum board, but it's worth it. Let's talk about why.

Why use testing frameworks

The great thing about running unit tests is that you can test your functions with many parameters all at once and it forces you to consider all possible test cases. This by the way is critical during an interview as this may come up, and goes hand in hand with the same creative thought process that created the algorithm in the first place.

Testing is an art

It's interesting that testing and debugging is not extensively focused on in undergraduate courses. It takes up a considerable amount of a real world developers time, especially while working on a complex component with little to no documentation on dependencies. Ouch! It could be because they just assume we'll learn it out on the job, which is true, but let's not dive down that rabbit hole.

Chai & Mocha

Chai and Mocha are the Ying and Yang of Testing in NodeJS. Not really, but it sounds better like that. Mocha is a library designed to execute tests and generate reports for you, while Chai is an assertion library which uses cool english like function strings to increase readability of the code.

Unit tests are meant to be quick and precise. This means that it's difficult to unit test a function that is doing too many things, or that is what is known as a bloated function. Here's an example.

Could you tell what this function does? Just by looking it's very hard to tell. It was originally developed to map our external API responsible for the blog's content to our ReactJS code format. As you can see this function has multiple parts to it. Although it's not as complicated as what could exist out there, it illustrates a good point in code which is to make it both readable and implement encapsulation. From a high level perspective, here's the workflow of the function. Wait until all code is rendered from the API -> dynamically add the parsed JS -> process all code tags -> render new formatting -> cleanup junk XML. If we write code which is comprised of simple functions then we can better unit test the function to ensure each individual part behaves properly. Here's the function rewritten.

const readable = async () => {
   wait codeAvailable;

   parseCodeTags();

}

Capitalizing on encapsulation is what separates the good developers from the great. Now that we've got the basic mechanics out of the way, it's time to write some tests.

Mocha

Mocha is a library used for organizing and displaying your tests in a quickly executable and visual way. There are two main functions used, being describe() and it(), which allow you to develop a hierarchy behind your tests. obviously, covering all the functions of this library and their use cases is out of scope but there's also a beforeAll() and beforeEach() function which allows you to, as you might expect, execute a block of code before all and before each test which would allow you to initialize the structures used in the tests and clean up any junk in those structures before the next test for example. Running the command 'Mocha', in say your root directory or putting it behind npm test and specifying your test folder will output a series of test results and detailed error diffs for viewing. Say we have two functions to test. One is getPosts(), one is processPosts(). Here's what our workflow might look like in this case. 

const assert = require('assert');

describe(' Testing function: getPosts()' , () => {    
   it(' gets all posts without returning an error HTTPS code ', ()
      assert(HTTPCodeValid);
   }); 
   it(' returns valid data from the call ', () => {
      assert(checkDataValid());
   });
});
describe(' Testing function: processPosts() '), () => {
   it(' processes all posts without throwing an error ', () => {
      assert(errorFree);
   }); 
   it(' returns properly formatted data with defined fields of proper type ', () => {
      assert(checkDataValid());
   });
});

The first thing to note is the output that is below. If an assertion ( or any error ), is thrown in one of the it() functions then that error will be nicely displayed with a description. In addition, we will see a diff between the expected result and actual result if encompassed in an assertion library. assert is native to NodeJS however there is a popular assertion library called Chai which is slightly more verbose but allows for more configuration in your tests.

Let's talk about Chai

In the code above we used the function assert() which will throw an AssertionError if it's contents are falsey and therefore will display the text within the it() function's first argument. It's pretty straightforward but lacking from a functionality perspective. If you're looking for a more readable way to write tests, check the values of return objects and their types, then Chai is perfect for your application development.

Chai has something called a language chain, which allows the code to be written as if it were plain english, I'm looking at you python contributors, and therefore makes the testing lines more readable. There are a couple different flavours of their assertion functions shown in the picture below.

There are a couple of things to highlight. One of them is the usage of language chains in both the Should() and Expect() modules such as to.be.a() or to.have.property(). The documentation outlines all these chains but from a functional perspective they are meaningless. The last property of the chain is the only property which constitutes any functionality behind it, and therefore have.to.property() would work the same as to.have.property(). This is great if you can formulate basic english sentences. There are many of these so called functional methods, such as checking the type, doing a deep comparison between objects, which is used to compare two objects for equality as === on objects won't work because NodeJS stores the addresses of objects which would be different, and many more. You can check to ensure that functions throw an error when they should with expect( foo ).to.throw( err ), or see if a returned object has a specific key with expect( foo ).to.have.property( 'key' ).

Some baseline unit tests to run on your application

In general, here are some key tests which are run in industry to ensure data integrity throughout an application. These points are by no means exhaustive but provide a good starting point for a simple application.

  1. Run tests which check the type of return value ( string, object, number, etc. )

  2. Run tests to check the function does not throw an error ( assuming proper error handling is in place

  3. Run tests with different types of inputs across the entire spectrum of possible inputs ( here's where the beneficial part comes in ).

  4. Create a .json test folder with all the test data, and a test factory which may map the data to the function if needed.

In the next blog post we'll outline the specific tests and general workflow used in writing those tests. This workflow is found by the team to be the most efficient at supplying a large variety of input data while writing minimal extra code. After all, we want the testing process to be as efficient as possible and not delay our product releases, yet still cause us to think about the types of inputs and outputs which are possible and therefore do a gap analysis at the function level before a customer ever sees the finished product. Stay humble.
