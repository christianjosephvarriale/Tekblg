How to Add Pagination to your site

  • 2020-04-17T00:00-04:00
  • By Tekblg

Your website is too slow

Let's talk statistics, if your site is delayed by 1-5 seconds in load time that increases the bounce rate probability by 90%. If you've got a site with a lot of content, maybe a blog site like this one, you can very quickly load in too much data when your application mounts, and send your users off to a reddit memes subreddit. Hey.. who knows?

One technique to increase page speed is pagination

Say you've got lots of data to potentially show in a single component.

Google Lighthouse and SEO

Wow you've got a perfect blog already, time to apply for google adsense. I digress, this simple approach is fine is you have a couple of posts with minimal content. What if however, you want a good blog site with actual visibility? Crazy ask right... If that's the case, maybe you want high res images to captivate people, maybe you want a .MP4 video as a thumbnail. Great idea Scotty, one issue, google lighthouse will give you numbers like this.

Maybe that's not a big deal for you that nobody, not even your best friend wants to wait 15 seconds for your site to load. Primarily because:

  1. The user might think the page has stalled, and would therefore continually refresh the page

  2. You've lost credibility as a professional site, if your page doesn't load, your dev opinions are irrelevant

How to increase initial loading speed

There are various ways to do this, their combination resulting in the best performance, but from a cost benefit analysis here is the list in order

  1. Pagination for a blog style site or a site

  2. Image compression into optimal formats and sizes

  3. Lazy Loading

Pagination is particularly important to implement on a blog because the requested JSON file can be quite large. You can view if the .JSON payload is the bottleneck to an application by viewing the app in the development tools

The meta request taking 797ms to load is a .JSON file. If that file were smaller, the time to load might be a fraction of that, an ideal range would be ~ 30kb. In total, a page should not load more than 1 MB of information. If it has to, then the page will be slow. This is exaggerated on a mobile network, which tends to run on 3G data and therefore can't download as much data at a timely rate, thanks bandwidth restrictions... By using the network section of google dev tools you can diagnose where is your application's weaknesses occur in a way google pagespeed insights can't. It'll also be used when points 2 & 3 are explained in following posts... thanks for keeping us waiting, the suspense is killing me

Pagination Defined

So we've determined that a .JSON file with 550 posts is too large. By definition, Pagination is the process of spreading out data to multiple pages so that each page is both more lightweight and the user has less scrolling to do. From a revenue perspective, that's also more pages to visit on a site, meaning more opportunities for users to see ads and generate cash for you. It's why spammy sites post content in 30 page slideshows... Genius

What not to do

Let's put a limit on the number of posts we use at any given time. One could do this from the front end, meaning we take our Array of Objects: Posts[0..N] = [{},{},...,{}] and we slice from the 0th index to the Nth for a total of N posts for example. This however doesn't necessarily speed up load time as much as it could. It would, allow for less Posts to be rendered to the screen, but not decrease the time to load the Posts.json file from the server. Thus a better way to do it is to contain data for only N number of posts in that GET server request

Rails and routing to pages

This is not an in-depth ruby on rails tutorial. We will assume here you know some basic syntax behind routing, and controllers

Here we will setup a controller on our API app which will receive the param[:page] and based on that determine to slice our posts accordingly. Note: be sure to sort your posts based on how you'd like them to render, with the most recent one first LIFO

routes.rb

Rails.application.routes.draw do  
     get '/posts/:page', to: 'posts#index'
     get '*path', to: "application#fallback_index_html", constraints: ->(request) do    !request.xhr? && request.format.html?  
   end
end

posts_controller.rb

class PostsController < ApplicationController

    # GET all posts based on page    
    def index
       srt_pos = ( params[:page].to_i - 1 ) * 12
       end_pos = srt_pos + 11
      @posts = Post.all.order(created_at: :desc)[srt_pos..end_pos]
      render json: { posts: @posts, length: Post.all.length }
    end

    private      
       # Only allow a trusted parameter "white list" through.
      def post_params    
         params.require(:page) 
      end
end

To create the API to our backend system, we need a URL structure that will allow the backend to know:

  1. What resource we are requesting

  2. In our case for pagination, there needs to be a synchronized way to determine which posts to send

We could either request for a specific range of posts such as specifying a range of ID's, we could effectively make a POST call with a SQL where clause for example getting all posts whose DateTime is equal to Date.today, or if the maximum number of posts you'd like to display is fixed, then we could go ahead and request a page. That's the approach we're taking here as the logic is clean

We have the get '/posts/:page', to: 'posts#index' which allows us to both upload a param[:page] without making a post call. You could also make this request using a query string, such as GET '/posts?page=N', which is fine as well but rails makes our syntax a little cleaner and clearly defined in the routes.rb file

Finally, our posts controller takes the param[:page] and converts it to an integer, does the array slicing based on the ordered Post.all array, in our case this ordering is by DESC Datetime, and rendering the length of the total array to let the front end know how many pages in total exist

Pagination.js

Before we show you the code, it's important to speak more about the system architecture. WIth our current URL setup, when users click on your blog, the front end needs to know which page to request from the server. One way to do that is by default forward users to 'yourdomain.com/blog/page/1' and therefore the request to the backend can simply take the last number of the URL

class Pagination extends React.Component {  
   render() {    
      const props = this.props;
      const url = window.location.href;    
      let page = url.slice(url.lastIndexOf('/')+1).replace(/\D/g,'');   
      let backDisabled = false;
      let forwardDisabled = false;
      const totalPages = Math.floor(props.total / 12) + 1;

      // on the first set of pages
      if ( Math.floor(( currentPage-1 ) / 5) == 0 ) {
         backDisabled = true;
      }

      // on the last set of pages
      if ( totalPages - (Math.floor((currentPage-1) / 5) * 5) <= 5) {        
         forwardDisabled = true;
      }

      const STARTING_INDEX = (Math.floor((currentPage-1) / 5) * 5);
      const END_INDEX = STARTING_INDEX + 5; 

      const pageArray = [...Array(totalPages + 1).keys()].slice(1,).slice(STARTING_INDEX,END_INDEX)
      
       const pageElements = pageArray.map((pageNumber) => { 
          if (pageNumber == currentPage) {
             return  <li><span className={[styles.pgnNum,styles.current].join(" ")}{pageNumber}</span></li>        
             } else {        
             return <li><Link className={styles.pgnNum} to={`${pageNumber}`}>{pageNumber}</Link></li> 
             } 
        })
        
        let back;   
        if (!backDisabled) { 
           back = <li><Link className={styles.pgnPrev} to={`${(parseInt(currentPage) - 5)}`}>Prev</Link></li>
        }
        
        let forward; 
        if (!forwardDisabled) {
           forward = <li><Link className={styles.pgnNext} to={`${((totalPages - currentPage > 5) ? parseInt(currentPage) + 5 : totalPages)}`} >Next</Link></li>
        }
    
        return (
            <div className={[styles.row,styles.paginationWrap].join(" ")}>
               <div className={styles.colFull}>
                  <Router forceRefresh="true">
                     <nav className={styles.pgn}>
                        <ul>
                            {back}
                            {pageElements} 
                            {forward}  
                      </ul>
                      <div>showing elements {(parseInt(currentPage-1) * 12) + 1} - {(((parseInt(currentPage) * 12) + 1) > props.total) ? props.total : ((parseInt(currentPage) * 12) + 1)} out of {props.total}
                      </div> 
                   </nav>   
                </Router>  
            </div>    
        </div>  
      );  
   };
};
export default Pagination

Lot's of code here, but let's walk through it in a general workflow

  1. First we take the params[:page] from the url, and filter out any junk that's added to the end like the # symbol

  2. We determine the total number of pages based on how many posts we want to show, in this case 12

  3. We have to decide how many potential pages to jump to we'd like to show at a given time. For example, should our pagination show pages 1 - 5? with arrow keys on the ends for moving by a single page? 5 pages? should the highlighted page always be in the middle of the pagination except for the initial pages? all of these design questions must be answered. In our case, we have a maximum of 5 pages that will be shown at once, with arrows advancing by either 5 pages in either direction, or the maximum amount that exists. For example, if I'm on page 4 and only 3 more pages exist, the NEXT button will take me to the last page instead of non-existent page 9

  4. We then format the page links based on a <ul><li>...<li></ul> type structure

  5. This next step is mainly for SEO, but if the back or forward Nav buttons don't exist, they shouldn't even be rendered. If they are rendered but hidden using display:none then a search engine may index that page, leading to poor google ranking since that page doesn't exist

  6. Finally we return the component, with the back button, elements, and forward button visible. It's helpful to allow give the user knowledge of how many elements are on the page, possibly with a ID ranking them for taste, and how many elements or pages exist total.
